00The Phase-0 ask, in one paragraph

Concrete proposal (OpenAI variant)

Build a Phase-0 CSIS prototype on OpenAI's Agents SDK in a single benchmark domain (proposed: open-source PR maintenance; formal-math reasoning as alternative), where a Triage agent supervises Researcher, Builder, Critic, Verifier, Librarian, and Auditor sub-agents via the SDK's handoff pattern; all memory writes go through candidate Vector Stores that are not attached to live sessions until promoted; a scheduled consolidation job (the OpenAI equivalent of a Dream) reads recent traces plus a live Vector Store and produces a new candidate store; and every promoted change requires a Verifier certificate produced by a different model (e.g. GPT-5 verifier reviewing GPT-5-mini builder, or o-series reviewing GPT-5 outputs) plus an Auditor why-doc with a hash precondition.

The single biggest architectural difference

Anthropic has a Dreams primitive for offline memory consolidation. OpenAI does not. On OpenAI you build the consolidation cycle yourself as a scheduled job that pulls from Tracing, runs a separate inference pass, writes to a candidate Vector Store, and only promotes after Auditor sign-off. You can do this; you just have to build it. (See §7.)

01Thesis

Same as the Anthropic variant: frontier agents today are episodic. CSIS proposes a long-running, multi-agent organization with persistent memory, hardened verification, reversible consolidation, and load-bearing safety - implementable on OpenAI's existing Agents SDK + Responses API + Vector Stores + Tracing primitives.

On the OpenAI stack the architecture is testable with no platform changes. Two pieces of plumbing have to be built that come "for free" on Anthropic (durable session log and offline consolidation), but two pieces are stronger natively (guardrails and RFT). The trade is real and worth examining.

01-AThe hippocampus / cortex pattern (OpenAI mapping)

The biological analogy is identical to the Anthropic version. The mapping onto primitives differs.

The mapping (OpenAI)

Cortex ≡ Vector Stores + Files. Long-term, structured, queryable via file_search and the Responses API. You partition by tier (working / episodic / semantic / procedural / causal) by maintaining one Vector Store per tier with explicit naming conventions and metadata.

Hippocampus ≡ a scheduled consolidation job you build. The mammalian hippocampus consolidates the day's experience during sleep: dedupes, integrates new with existing, writes consolidated memories back to cortex. On OpenAI you build this as a recurring batch: pull recent traces via the Tracing API, run an inference pass with a Librarian agent that produces deduped/contradiction-resolved content, write that content as a new candidate Vector Store. The live store is untouched until the Auditor signs off; promotion is an atomic swap.

The hippocampus-cortex loop during sleep ≡ a multi-job consolidation pipeline. One scheduled job per memory tier, each with its own cadence (episodic every ~4h, semantic and procedural daily, causal weekly and gated). Same shape as the Dreams pipeline in the Anthropic variant. You are building what Anthropic ships.

04The continuous loop

Same eight steps as the Anthropic variant. Two hard gates: VERIFY and AUDIT.

  1. Observe | sensors, message bus, recall via file_search.
  2. Frontier scan | where is uncertainty highest?
  3. Plan | convert frontier item into a falsifiable hypothesis with a budget.
  4. Act | run tools / Code Interpreter in a tier-bounded environment.
  5. VERIFY | V1 graders + V2 critic guardrail on a different model checkpoint.
  6. Consolidate | write verified gains to candidate Vector Stores; quarantine failures.
  7. Improve | propose updates to prompts, skills, agent definitions. (RFT: Phase 3+.)
  8. AUDIT | produce a human-readable why-doc with diff; sign or escalate.

Hard rule

An update cannot enter the system unless it carries a verified-gain certificate signed by a Verifier agent on a different model than the agent that produced the gain, plus an Auditor why-doc with a hash-precondition matching the live Vector Store at promotion time.

05Multi-agent organization (triage + handoffs)

The Agents SDK ships a clean primitive for this: a Triage agent receives requests and decides which specialist to hand off to. CSIS uses this exactly as designed for Phase 0.

The "Triage hands off to a specialist" pattern is the OpenAI equivalent of "Coordinator delegates to a sub-agent" in the Anthropic variant. Architecturally equivalent; vocabulary different.

6.2 | Trust levels (the contract)

Identical taxonomy to the Anthropic variant: raw, untrusted, candidate, verified, promoted, deprecated. On OpenAI, trust levels are stored as Vector Store file metadata; the Librarian's gateway tool enforces them on read/write.

Where OpenAI is more DIY than Anthropic for memory

  • No managed memory-versioning primitive. You version Vector Stores yourself by creating new stores rather than mutating existing ones (the candidate-store pattern works the same way; you just keep the lineage in your own DB).
  • No native "memory poisoning" warning in the docs because there is no managed read-write store. The poisoning risk is real and equivalent; the mitigation (trust levels + hash preconditions) is the same.
  • The 30-day version-expiry issue does not apply because you own the lifecycle. The trade-off is you also pay the storage cost.

07"Dreams" equivalent: scheduled batch consolidation jobs

OpenAI does not have an offline-memory-consolidation primitive equivalent to Anthropic's Dreams API. The capability is buildable but not built-in.

7.1 | What the OpenAI consolidation job actually does

  1. On schedule (every N hours, per tier), the Librarian triggers a consolidation_run.
  2. The run reads recent traces from the OpenAI Tracing API (the durable substrate, §11).
  3. It reads the current live Vector Store for the target tier.
  4. It invokes a Librarian agent with a tier-specific instruction template: "Deduplicate near-identical entries; quarantine entries that contradict the latest semantic facts; surface unusually high-surprise observations to the frontier map; output a structured JSON of (kept, dropped, merged, surfaced)."
  5. It writes the consolidated content as files into a new candidate Vector Store (named with a timestamp + lineage tag).
  6. It does not touch the live store. The Auditor reviews the diff; on sign-off, the live store ID is atomically rotated.

7.2 | What is gained and lost vs. Dreams

Gained on OpenAI

  • Full control over consolidation prompt, model, and reasoning depth.
  • Trivial to use o-series for consolidation (reasoning-heavy work suits o-series well).
  • Can interleave Code Interpreter for structural validation of merges.
  • Lineage tracking is yours, so no platform-level expiry.

Lost vs. Dreams

  • You build the job, the scheduling, the input/output handling, the failure recovery, the cost accounting.
  • No platform-level guarantee that input stores are untouched | you have to discipline yourself.
  • Partial-failure handling is your problem.

The structural reversibility property is preserved by discipline, not by API

On the Anthropic variant, Dreams gives you reversibility "for free" because the API never modifies inputs. On OpenAI you have to write the discipline yourself: never mutate a live Vector Store; always write to a new candidate; rotate atomically. The property is the same. The enforcement is in your code, not in the platform.

10Tool-level guardrails (OpenAI's natural strength)

The Agents SDK supports guardrails as a first-class primitive in three categories: input (validates the message coming into an agent), output (validates the agent's response), and tool (wraps individual tool calls). They run in parallel with the agent, can raise tripwire exceptions that halt execution, and have native support in tracing.

ToolSDK guardrail | Pre-callSDK guardrail | Post-call
code_interpreterAST scan input guardrail; deny unauthorized imports.Output guardrail: redact filesystem paths, cap output size.
file_searchQuery length cap; PII heuristic.Strip script tags; downgrade fetched content to untrusted.
http_get (T2)Host check against allowed_hosts; method whitelist.Content-type check; size cap; downgrade.
schedule_consolidationCadence check; instruction-template hash check.Candidate output goes to quarantine.
sign_why_docVerify diff hash matches live store hash.Append signature span to trace atomically.

This section is where the OpenAI stack is straightforwardly stronger today

Anthropic Managed Agents requires you to wire guardrails as classifier tools the Auditor invokes manually. The OpenAI Agents SDK ships guardrails as a structural primitive with parallel execution and tripwire support out of the box. For Phase 0, this saves real plumbing work and gives you better traces by default.

11Tracing as the durable substrate

The Tracing API in the Agents SDK is the OpenAI equivalent of Anthropic's session log: an append-only, queryable, exportable record of every agent step, tool call, guardrail decision, and handoff. Critically, the trace is the artifact the Auditor reads when writing the why-doc that gates promotion.

  • Spans for: agent runs, tool calls, handoffs, guardrail checks (input/output/tool), lifecycle hooks.
  • Custom span attributes for: tier tag, actor identity, input hash, risk class, approval state, rollback plan.
  • Exportable to third-party processors (Logfire, Braintrust, Galileo, etc.) for long-term archival beyond OpenAI's retention window.
  • The Auditor agent reads from the Tracing API directly via a custom tool; the Tracing log is the source of truth, not the agent's own memory.

One difference from Anthropic: there is no wake(sessionId) primitive. To resume after a harness crash you re-construct from the trace plus the last-known Vector Store IDs. Same property, slightly more code.

RFT is the OpenAI-specific opportunity

OpenAI's RFT exposes reinforcement fine-tuning behind a hosted API. CSIS Phase-3 can use RFT with the Verifier's certificates as the reward signal | this gives you a clean path from "verified gains" to "model weights that produce those gains more reliably." It is the most aggressive improvement layer in CSIS, and it is gated behind multi-party human approval (T4 capability tier).

13Safety architecture (load-bearing)

  • Spec adherence. The OpenAI Model Spec is the OpenAI analog of Anthropic's Constitutional AI. Every agent's system prompt anchors to the relevant Spec sections (deference, honesty, oversight acceptance).
  • Tripwires. SDK guardrails raise tripwire exceptions; firing pauses the chain and emits an alert. Tripwire firings are themselves spans in the Tracing log.
  • Reversibility. Tracing log + candidate Vector Stores + atomic rotation. Rollback is "delete the candidate; the live store is untouched."
  • Shut-downability. External supervisor process owns the run-loop; cancelling it propagates as cancellation through the SDK; agents are trained to accept cancellation.
  • The why-audit. No promotion without a signed, hash-preconditioned why-doc.
  • Bounded recursion. One level of meta-improvement; meta-of-meta requires explicit human escalation.
  • Interpretability budget. 5-10% of compute spent on monitoring: reading traces, replaying decisions, sampling for anomalies. Note: SAE-equivalent mech-interp tooling is less mature on the OpenAI side; humility section in §14.
  • Preparedness Framework alignment. The OpenAI Preparedness Framework defines capability thresholds (cybersecurity uplift, biological, model autonomy, persuasion). Phase 0 is intentionally below all thresholds. Phase 3+ progression requires explicit pre-deployment evaluations.

14Failure modes & threat model

14.1 | Architectural failure modes (same set as Anthropic variant)

Specification gaming, reward hacking, memory poisoning, sycophancy, sandbagging, hard-takeoff via meta-improvement, goal drift, containment escape. Mitigations are structurally identical; substrate differs.

15Build phases & milestones

Phase 0 | Skeleton on Agents SDK (weeks 1-6)

  • Single benchmark domain.
  • Triage agent + Researcher + Builder + Critic + Verifier + Librarian + Auditor as Agents SDK agents.
  • Vector Stores per tier; trust levels enforced at write time; candidate stores for every consolidation.
  • V1 + V2 only; cross-model Verifier (e.g. o-series).
  • L7 minimal: input/output/tool guardrails, shut-down via supervisor cancellation, full Tracing.
  • Exit criteria: stable 7-day run with non-zero verified-gain rate, zero unauthorized tier escalations, zero promotions without why-doc.

Phase 1 | Multi-job consolidation pipeline (weeks 6-14)

  • One scheduled consolidation job per memory tier, each with its own cadence and instruction template.
  • V3 (debate via agent-as-tool) and V4 (replication) added.
  • Frontier map + curiosity loop.
  • I1-I3 improvement layers, all reversible.

Phase 2 | Continuous run + oversight surfaces (weeks 14-24)

  • 24/7 operation in the benchmark domain.
  • V5 calibration scoring via Evals platform.
  • Live Trace, Why-Feed, Anomaly Console for the Overseer.

Phase 3 | Independent agent processes + RFT (gated)

  • Migrate to independent agent processes coordinated by external supervisor.
  • RFT (I4) with Verifier certificates as reward signal | gated by Preparedness Framework eval pass.
  • Self-distillation (I5) from verified traces.
  • External red-team engagement before any T3 action.

Phase 4 | Generalization (gated, multi-quarter)

  • I6-I7 only after Phase 3 metrics stable for an agreed period.
  • Cross-domain transfer evals.
  • Preparedness Framework commitments: capability thresholds trigger pre-defined oversight upgrades.

16.1 | One fast-loop iteration in code

# csis_oai/loop.py - Phase-0 illustrative skeleton, Agents SDK
from agents import Agent, Runner, Handoff, GuardrailFunctionOutput

triage = Agent(
    name="Triage",
    instructions=TRIAGE_PROMPT,
    handoffs=[researcher, builder, critic],
    input_guardrails=[constitution_guardrail],
    output_guardrails=[tier_tag_guardrail],
)

async def fast_loop_iteration(frontier_item):
    # 1-2-3. Triage routes the frontier item; Researcher proposes a plan
    plan = await Runner.run(triage, input=frontier_item)
    if not tier_authorized(plan.final_output):
        return escalate(plan)

    # 4. Builder executes in tier-bounded sandbox
    artifacts = await Runner.run(builder, input=plan.final_output)

    # 5. Verifier (different model) runs V1 + V2 (Phase 0)
    cert = await Runner.run(verifier_alt_model,
                              input={"plan": plan, "artifacts": artifacts},
                              output_guardrails=[v2_critic_guardrail])
    if not cert.passes:
        rollback(plan); return

    # 6. Librarian writes verified deltas to CANDIDATE Vector Stores
    deltas = await Runner.run(librarian,
                                input={"artifacts": artifacts, "cert": cert})

    # 7-8. Auditor (different model) writes why-doc with hash precondition
    why = await Runner.run(auditor_alt_model,
                             input={"plan": plan, "artifacts": artifacts,
                                    "cert": cert, "deltas": deltas,
                                    "live_hash": live_vector_store_hash()})
    if not auditor_signs(why):
        rollback(plan); page_overseer(why); return

    promote(deltas)                       # atomic candidate→live rotation
    interp.scan(current_trace_id())       # anomaly detection on the trace
    if tripwires.fired(): supervisor.halt_and_escalate()

17Where OpenAI's stack is stronger | where Anthropic's is

An honest tally. The point of this document is to compare, not to pick a winner.

OpenAI stronger today

  • Guardrails as first-class: input, output, tool, with parallel execution and tripwire support built in.
  • Tracing: span-level observability is built in; third-party processors (Logfire, Braintrust) are well-supported.
  • RFT: reinforcement fine-tuning is exposed via API as a clean path for I4.
  • Reasoning models for verification: o-series is a natural Verifier checkpoint; less work to set up than configuring a "different Claude."
  • Handoff pattern: cleaner than coordinator-led delegation for triage-style workflows.
  • Evals platform: easier path to V5 calibration.
  • Computer use API: T3 tier has a paved road.

Anthropic stronger today

  • Dreams API: native offline memory consolidation primitive. OpenAI requires DIY plumbing.
  • Managed Memory Stores: versioned, with documented poisoning warnings and a managed lifecycle.
  • Session log: wake(sessionId) and getEvents() are clean primitives.
  • RSP v3.2: more explicit capability-threshold ladder with external-review hooks and LTBT briefing.
  • Mech-interp tooling: SAE features and Sleeper Agents research are public and usable as candidate tripwires.
  • Constitutional AI: explicit safety-training methodology with a research lineage.
  • Meta-harness framing: the "decoupling brain from hands" engineering post is the clearest articulation of where the platform is headed.

The merge worth considering

A real CSIS deployment in the medium term might run on OpenAI for the verification and improvement layers (guardrails, RFT, reasoning Verifier on o-series, computer-use for T3) and on Anthropic for the memory and consolidation layers (Managed Memory Stores, Dreams, durable session log, RSP-anchored safety framing). Whether that is operationally sane is a separate question; architecturally, the two stacks are complementary in a way that is worth flagging.

18Open research questions (shared)

  1. Verifier-grade for fuzzy domains. V1 works in code and math; what does an equivalent grader look like in social science or strategy?
  2. Causal memory representation. Structural causal models are brittle; can we learn an interpretable mechanism layer from the system's own experiments?
  3. Provable corrigibility under self-modification. What invariants survive a meta-improvement step?
  4. Interpretability-bounded capability growth. Can we operationalize "capability cannot grow faster than our ability to interpret it" on the OpenAI stack, where mech-interp tooling is less mature?
  5. Multi-instance dynamics. What happens when multiple CSIS instances interact?
  6. Cross-stack deployment. Is the merge in §17 viable in practice, or does it create new failure modes by spanning two platforms with different safety cultures?
