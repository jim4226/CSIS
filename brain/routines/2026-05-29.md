### deferred — Dynamic Workflows multi-agent orchestration pattern (v2.1.154)
- URL: https://code.claude.com/docs/en/changelog
- Source: code.claude.com/docs/en/changelog, v2.1.154 published 2026-05-28
- Theme: 1 — Multi-agent coordination
- Touches: `csis/agents/coordinator.py` (`run_continuous()`, potentially a new `run_parallel()`)
- Size: medium (≤500 LOC — new method + EventLog concurrency handling + BudgetTracker parallel reservation)
- Risk: medium — close to coordinator architecture; the H6 note in `Coordinator` already anticipates this but a concrete implementation would touch the run loop
- Design sketch: Add `run_parallel(frontier_items, *, max_concurrent: int = 4)` to `Coordinator` as an alternative to the synchronous `run_continuous()`. Each item is assigned its own `MemoryHierarchy` slice so the single-writer lock invariant holds at the slice level, not the global level. A `ThreadPoolExecutor` dispatches up to `max_concurrent` `run_iteration()` calls. Results are collected and the Auditor receives a merged batch cert. The `EventLog` already handles concurrent writes via the cross-process file lock (Snapshot-12); `BudgetTracker` needs a reservation-for-N primitive. Regression test must fail before the change (concurrent calls to the current `run_continuous()` raise under `sys.setswitchinterval(1e-9)`) and pass after.
- Why deferred: medium complexity; the coordinator architecture comment explicitly defers this to "the Anthropic scheduler (≤25 concurrent threads) would replace this in a real deployment." This PR would be a CSIS-prototype approximation of that scheduler. Needs cross-cycle design discussion before code.
