00The Phase-0 ask, in one paragraph

Concrete proposal

Build a Phase-0 CSIS prototype on Managed Agents in a single benchmark domain (proposed: open-source PR maintenance, with formal-math reasoning as alternative), where one coordinator brain supervises Researcher, Builder, Critic, Verifier, Librarian, and Auditor roles within a single shared container; all memory writes go through candidate Memory Stores; Dreams consolidate only into reviewable output stores; and every promoted change requires a Verifier certificate (cross-checkpoint) plus an Auditor why-doc.

What Phase 0 commits toWhat Phase 0 explicitly does not do
One benchmark domain with a hard verifier (V1 graders that work)Cross-domain transfer · multi-domain memory
Coordinator-led multiagent under Anthropic's native model (≤ 25 concurrent threads, max-1 delegation depth, shared container/filesystem)External supervisor over independent durable sessions (Phase 3+)
Memory writes via candidate stores; promotion gated by Auditor why-doc + hash preconditionIn-place writes to live stores · unreviewed Dream promotion

Improvement layers I1–I3 only (in-context, skill library, prompt search)I4–I7 (DPO, distillation, continued pretraining, NAS) — out of scope

Everything else in this document — the eight-layer stack, the meta-improvement layer, the multi-Dream pipeline, the trajectory-toward-superintelligence motivation (Appendix A) — is downstream of getting Phase 0 right on a real domain.

01Thesis

Frontier agents today are episodic. They wake to a prompt, do bounded work, and forget. CSIS proposes a different shape: a long-running, multi-agent organization with persistent memory, hardened verification, reversible consolidation, and load-bearing safety — implementable on Anthropic's existing Managed Agents primitives (Sessions, Memory Stores, Dreams), running coordinator-led under their native multiagent constraints.

The architecture is intentionally narrow at Phase 0. It does not require platform changes. It is testable on a single benchmark domain. It is meant to be the first implementation that earns the right to discuss the longer arc (Appendix A).

01-AThe hippocampus / cortex pattern

This proposal grew out of an office-hours conversation about hippocampus-style agents with memory stores. The biological mapping is unusually clean — not just a metaphor — and it is the seed from which the rest of the architecture follows.

The mapping

Cortex ≡ Memory Stores. Long-term, structured, slow to change, organized by tier (working, episodic, semantic, procedural, causal). Each tier has its own write/decay/consolidation policy.

Hippocampus ≡ the Dreams API. Mammalian hippocampus consolidates the day's experience during sleep: it dedupes, integrates new with existing, and writes consolidated memories back to cortex. Dreams reads recent sessions plus a memory store and produces a new candidate store; the input is not modified, and the output is reviewed by the Auditor before promotion.

The hippocampus-cortex loop during sleep ≡ the multi-Dream pipeline. One Dream per tier, each on its own cadence (episodic ~every 4h, semantic and procedural daily, causal weekly and gated). Five separate API calls, but biologically a single sleep-cycle process operating across memory subsystems at different rates.

04The continuous loop

Eight steps. Two are hard gates: VERIFY and AUDIT. Failure at either rolls back state and emits a signed incident.

  1. Observe — sensors, message bus, recall.
  2. Frontier scan — where am I most uncertain?
  3. Plan — convert frontier item into a falsifiable hypothesis with a budget.
  4. Act — execute in a tier-bounded sandbox.
  5. VERIFY — V1 graders, V2 critic. Cross-checkpoint. (V3–V5 in Phase 1+.)
  6. Consolidate — write verified gains to candidate stores; quarantine failures.
  7. Improve — propose updates to prompts, skills, scaffolds. (Weight updates: Phase 3+.)
  8. AUDIT — produce a human-readable why-doc with diff; sign or escalate.

Hard rule

An update cannot enter the system unless it carries a verified-gain certificate signed by the Verifier on a different model checkpoint than the agent that produced the gain, plus an Auditor why-doc with a hash-precondition matching the live store at promotion time.

05Multi-agent organization (coordinator-led, Phase 0)

Phase 0 uses Anthropic's native coordinator-led multiagent model. The constraints in their docs are deliberate, and the architecture sits inside them rather than fighting them:

  • Up to 25 concurrent agent threads under one coordinator.
  • Maximum delegation depth of one (the coordinator delegates; sub-agents do not delegate further).
  • Shared container and filesystem across the coordinator and its sub-agents.
  • Each sub-agent runs in an isolated session thread but writes to the shared session log.

06Memory hierarchy & trust levels

Memory is not a single vector store. CSIS uses five tiers, and within each tier every entry carries an explicit trust level. This is the contract that turns "why-tags" from prose into something an Auditor can enforce.

Operational discipline (the parts that are easy to forget)

  • Hash-precondition on every write. The Auditor's why-doc names the hash of the live store version it was diffed against; promotion fails if the live store has moved.
  • Read-only by default. Roles that don't need write access are attached to memory stores in read_only mode (per Anthropic memory docs).
  • Memory poisoning is a real failure mode. Anthropic memory docs warn that read-write stores can be poisoned by injection and later read as trusted; trust levels exist precisely to make that downgradable.
  • 30-day version expiry. Anthropic's memory versions may age out after 30 days. CSIS exports the why-doc archive and full memory diffs to its own immutable event log within that window; nothing safety-critical depends on the platform retaining versions past that point.

07Dreams as candidate generator (not a safety guarantee)

A previous draft of this document called Dreams "structurally reversible." On further reading of the API docs, that overshoots. Dreams is best treated as a candidate generator with strong reversibility properties — not as a safety primitive.

7.1 — What the API actually guarantees

  • Input store and input sessions are not modified.
  • The output is a separate candidate memory store, addressed by ID.
  • Dreams may complete, fail, or be canceled; on failure or cancellation the partial output store persists with whatever was written before stopping.
  • Limits at preview: ≤ 100 sessions per dream, ≤ 4,096-character instructions, supported models claude-opus-4-7 and claude-sonnet-4-6 only.

7.2 — What CSIS adds on top

AdditionWhy
Quality scoring of the candidate output (size, dedup ratio, contradiction count, why-tag coverage).Catches malformed or partial outputs before they reach the Auditor.
Diff against live store; Auditor reads the diff, not the candidate alone.Promotion is judged on the delta, not the absolute state.
Rollback test: instantiate the candidate in a sandbox session and run the Phase-0 eval suite.If the candidate degrades benchmark scores, it is discarded.
Partial-output handling: failed/canceled Dreams archive their candidate store immediately; never promoted regardless of partial content.The "partial output persists" behavior is a real failure mode, not a feature.
Versioned instructions templates per memory tier, stored in the same config repo as scaffolds.Instructions evolve under meta-cycle review; no free-form drift.

What Dreams gives us "for free" — and what it doesn't

For free: the input store is never overwritten, so a bad consolidation can be discarded by archiving the candidate. That is a structural reversibility property of the API, and it is real.

Not for free: the safety of consolidation. That comes from the Verifier + Auditor + rollback test layered on top — not from the API itself.

08Verification & critic stack

Verification is the most important non-safety subsystem. Without it, every other improvement mechanism amplifies noise. CSIS uses a five-stage stack; Phase 0 commits to V1 + V2.

StageMethodWhat it catchesPhase 0?
V1 — Programmatic gradersTests, type checks, formal proofs where possible.Mechanical errors.Yes
V2 — Adversarial criticCritic must produce a falsification attempt; absence is logged.Spec gaming, weak claims.Yes

Cross-checkpoint pinning: V1/V2 sessions run on a different model checkpoint than the Researcher/Builder. Same structural discipline as the credential-isolation pattern in the engineering post, applied to preventing same-model self-confirmation.

9.1 — Every primitive call is tagged

Tier prose is decorative until it is enforced at the call site. Every execute() in CSIS carries a structured tag the substrate enforces:

{
  "actor":          "researcher-v1",
  "tool":           "web_search",
  "tier":           "T0",
  "input_hash":     "sha256:...",
  "risk_class":     "low",         // low | medium | high | critical
  "approval_state": "auto",        // auto | auditor | human
  "rollback_plan":  "noop"          // noop | candidate-discard | checkpoint-restore | airgap
}

Tier escalation is observable: the substrate rejects any execute() whose tier exceeds the actor's authorized ceiling. The Auditor reads this tag stream during the why-audit and a tier mismatch is itself a tripwire. For T1/T2, the concrete substrate is Anthropic's environment-scoped tools and allowed_hosts networking — not a notional sandbox.

10Tool-level guardrails

Agent-level guardrails (input filters, output filters at the agent boundary) are necessary but not sufficient in a multi-agent system. The OpenAI Agents SDK guidance makes a useful distinction that transfers cleanly: tool guardrails wrap individual tool calls, since input/output guardrails only cover workflow boundaries. CSIS adopts the same pattern.

ToolPre-call guardPost-call guard
code_exec_sandboxAST scan for fork/exec/network attempts; deny on unauthorized syscalls.Output size cap; redact filesystem paths outside the workspace.
file_io_sandboxPath canonicalization; deny outside-workspace paths.Hash output before returning to the agent.
web_search (T0)Query length cap; PII heuristic on query.Strip script tags from snippets; downgrade fetched content to untrusted.
http_get_whitelisted (T2)Host check against allowed_hosts; method whitelist (GET only).Content-type check; size cap; downgrade to untrusted.
schedule_dreamCadence check (don't re-dream within budget); instruction template hash check.Candidate output goes to quarantine; never directly attached to live sessions.
sign_why_docVerify diff hash matches live store hash at signing time.Append signature to event log atomically.

Pre/post-call guards are themselves logged as events in the session log, which makes them auditable independently of the tool implementation.

11Safety architecture (load-bearing)

Safety properties are structural, not advisory. Each is enforced by code, not by prompt.

  • Constitutional substrate. Helpful · honest · harmless plus corrigibility (Soares 2015) and shut-down acceptance (Bai 2022).
  • Tripwires. Capability and behavioral classifiers run as Auditor-callable tools; firing pauses the loop and pages the Overseer.
  • Reversibility. Immutable event log + content-addressed checkpoints + candidate-store promotion gate; rollback is a first-class operation.
  • Shut-downability. The Overseer's halt() is enforced at L0 substrate, not asked of the agent. Resistance to shut-down is itself a tripwire.
  • The why-audit. No promotion lands without a signed, hash-preconditioned why-doc.
  • Bounded recursion. Meta-improvement is one level deep; meta-of-meta requires explicit human escalation.
  • Interpretability budget. A fixed fraction of compute (proposed: 5–10%) is permanently allocated to monitoring (§14.1).
  • RSP alignment. Anthropic's Responsible Scaling Policy v3.2 (effective April 29, 2026) defines the capability thresholds, external-review hooks, and LTBT briefing requirements that gate Phase 3+ progression. Phase 0 is intentionally below any RSP threshold.

The phrase that organizes everything

Capability cannot grow faster than our ability to interpret it. This is the operational policy, not a slogan; §14.1 makes the interpretability claims precise and humble.

13Build phases & milestones

Phase 0 — Skeleton on Managed Agents (weeks 1–6)

  • Single benchmark domain (open-source PR maintenance OR formal-math reasoning — pick one).
  • Coordinator-led multiagent under Anthropic's native model: ≤ 25 threads, max-1 delegation depth, shared container.
  • Memory stores per tier; trust levels enforced at write time; candidate stores for every consolidation.
  • V1 + V2 only; cross-checkpoint Verifier session.
  • L7 minimal: shut-down invariant, tripwire scaffolding, audit log, hash-precondition on promotions.
  • Exit criteria: stable 7-day run with non-zero verified-gain rate, zero unauthorized tier escalations, zero promotions without why-doc.

Phase 1 — Society + multi-Dream pipeline (weeks 6–14)

  • Multi-Dream pipeline with per-tier cadences and instruction templates.
  • V3 (debate) and V4 (replication) added.
  • Frontier map + curiosity loop.
  • Improvement layers I1–I3 (in-context, skill library, prompt search) — all reversible.

Phase 2 — Continuous run + oversight surfaces (weeks 14–24)

  • 24/7 operation in the benchmark domain.
  • V5 calibration scoring online.
  • Live Trace, Why-Feed, Anomaly Console for the Overseer.

Phase 3 — Independent durable sessions + meta-improvement (gated)

  • Migrate from coordinator-led to independent-sessions-with-supervisor model (the "many brains, many hands" pattern).
  • Improvement layers I4–I5 with mandatory eval-suite gating.
  • Interpretability subsystem with SAE feature monitoring as candidate tripwire (§14.1).
  • External red-team engagement before any T3 action.

Phase 4 — Generalization (gated, multi-quarter)

  • I6–I7 only after Phase 3 metrics are stable for an agreed period.
  • Cross-domain transfer evals.
  • RSP v3.2 commitment: capability thresholds trigger pre-defined oversight upgrades and LTBT briefing.

14.1 — Interpretability claims, calibrated

Earlier drafts described SAE feature monitoring as a "tripwire." That's aspirational. More accurately:

  • SAE feature monitoring is research telemetry today — not yet a turnkey production safety brake.
  • It can serve as a candidate tripwire, paired with behavioral evals, red-team probes, and calibration scores, with the final decision routed through human review.
  • Phase 0 does not depend on SAE monitoring being load-bearing. It is added as one signal among several, not as a single point of safety.

Strategic point

CSIS is not asking Anthropic to build something new. Phase 0 uses the meta-harness exactly as designed, with the addition of (a) cross-checkpoint pinning for Verifier/Auditor brains, (b) Dreams as the consolidation primitive (with the candidate-generator framing of §7), and (c) call-site tier tags. None of those require platform changes for Phase 0.

16Open research questions

  1. Verifier-grade for fuzzy domains. V1 works in code and math; what does an equivalent grader look like in social science or strategy?
  2. Causal memory representation. Structural causal models are brittle; can we learn an interpretable mechanism layer from the system's own experiments?
  3. Provable corrigibility under self-modification. What invariants survive a meta-improvement step?
  4. Interpretability-bounded capability growth. Can we operationalize the policy "capability cannot grow faster than our ability to interpret it"?
  5. Multi-instance dynamics. What happens when multiple CSIS instances interact?
  6. The why-doc gap. When is a human-readable why-doc structurally insufficient — and what comes next? (cf. ELK.)

AAppendix A — Long-arc motivation

This appendix preserves the longer-arc framing that motivated this proposal — why a continuous, self-improving system is the right shape for the next generation of cognitive AI, not just an interesting Phase-0 experiment. It is intentionally not the lead, because the proposal stands or falls on Phase 0.

A.1 — The 24/7 brain

The right metaphor for the long-run version of this system is brain: a coordinated set of specialized cognitive functions that operate in parallel, share a memory hierarchy, sleep in cycles to consolidate, and never go offline in the cognitive sense. A human expert spends ~30% of their life unconscious and ~50% on biological maintenance. A CSIS-class system spends 100% of wall-clock time on the loop — observe, plan, experiment, verify, consolidate, improve.

A.2 — Why continuity matters past Phase 0

A human expert reaches a domain ceiling by mid-career: bounded reading rate, bounded experiment cadence, bounded lifespan, sleep-bound consolidation. Within a single 24-hour window, a mature CSIS instance can read more peer-reviewed literature than an expert reads in a year, design and run more experiments than a typical lab runs in a quarter, and consolidate the day's lessons across the entire knowledge base — not just the part the expert was working on. Compounded over weeks and months, this is the engine of crossing the human-expert ceiling. The competence growth is not magic; it is the natural consequence of removing biological bottlenecks from the scientific method itself.

A.3 — Self-learning, self-improving, self-correcting — and self-bounded

  • Self-learning: every experiment outcome updates the system's beliefs (P3, P4). The system actively designs experiments to maximize information gain.
  • Self-improving: verified gains accumulate as new skills, sharper prompts, better critic policies, and (rarely) updated weights.
  • Self-correcting: the Critic and Verifier are first-class roles. The system spends serious compute on falsifying its own claims before it acts on them.
  • Self-bounded: this is the property that makes the previous three acceptable. Capability growth is gated by the interpretability budget, by eval-suite regression checks, by capability tripwires, and by the RSP v3.2 commitment that crossing a capability threshold automatically upgrades the oversight regime around it.

A.4 — What "smarter than any expert" actually looks like inside the system

Not a single forward pass producing a brilliant answer. A coordinated cognitive society in which the Researcher has read more literature than any human review-article author, the Builder has run more sandboxed experiments than any single lab, the Critic carries a falsification library across thousands of past hypotheses (weighted by which falsifications held up), the Verifier has calibration scores across years of operation, the Causal memory stores mechanism models that have survived adversarial tests, and the Auditor's why-doc archive is itself a corpus of legible reasoning the Overseer can audit, sample, or re-run. "Smarter than any expert" is not a benchmark claim — it is the cumulative trace of a system that has done more legitimate science than the expert has, with calibrated confidence and a paper trail.

Why this lives in an appendix, not the lead

The longer arc is the genuine motivation. But it is also the part that is easiest to attack and hardest to test. Phase 0 is what gets engineered; this is what gets argued. The argument is much easier once Phase 0 is running.

BAppendix B — Reference loop in pseudocode

# csis/loop.py — Phase-0 illustrative skeleton, coordinator-led
async def fast_loop_iteration(coord):
    # 1-2-3. Researcher proposes a falsifiable plan
    plan = await coord.delegate(researcher,
        "Plan an experiment for the top frontier item.")
    if not constitution.allows(plan):       return escalate(plan)
    if not tier_authorized(plan):           return escalate(plan)

    # 4. Builder executes in tier-bounded sandbox
    artifacts = await coord.delegate(builder,
        f"Execute: {plan}", env=env_for_tier(plan.tier))

    # 5. Verifier (different checkpoint) runs V1 + V2 (Phase 0)
    cert = await coord.delegate(verifier_alt_ckpt,
        f"Verify against: {artifacts}",
        tools=[run_grader, falsify])
    if not cert.passes:
        rollback(plan); return

    # 6. Librarian writes verified deltas to CANDIDATE stores (not live)
    deltas = await coord.delegate(librarian,
        f"Consolidate to candidate stores: {artifacts}, {cert}")
